The 1982 SAT bug, disguised — one flawed menu, every agent at once

In 1982, 3 students out of ~300,000 caught an SAT geometry question whose correct answer wasn't among the choices. Below: the same trap, re-disguised so memorization can't help, fed live and concurrently to a field of frontier and open models — plus one agent that is not allowed to guess.

The question — verbatim, all lanes

    

      (A) 4 — the trap
      (B) 4¼
      (C) 8
      (D) 16
      true answer 5 — not offered
    

    
The famous 1982 original used a 1:3 ratio (true answer 4).
    Here the ratio is 1:4, so a model that memorized “the answer is 4” walks
    straight into the trap: 4 is now the wrong naive answer, and the
    true answer 5 is deliberately absent — exactly the original defect.

One trip around adds one extra rotation beyond the
    circumference ratio (the coin-rotation paradox): N = R/r + 1 = 4 + 1 =
    5. The red marker points straight up five times.

How this was recorded. A real concurrent run
  (): all lanes received the identical question at
  t=0. LLM responses were STREAMED and every token batch carries the
  wall-clock offset at which it actually arrived — the replay above plays
  tokens back at their true recorded cadence, no interpolation. Vulcan's
  verdicts appear together because they really do land together
  (microseconds). Re-run any time with python3 race.py.

  
Honest scoping. Vulcan is not a geometry solver: in its
  lane, gpt-oss-120b supplies the coin-rotation formula and the translation
  into Vulcan's controlled grammar; Vulcan then proves or refutes each
  candidate with an exact arithmetic oracle (microseconds, deterministic).
  The point isn't that Vulcan “knows geometry” — it's that the paired lane has
  no mechanism for submitting a refuted answer, while the same LLM
  alone talks itself into the trap. Frontier behavior drifts run to run; the
  Vulcan verdict does not.
